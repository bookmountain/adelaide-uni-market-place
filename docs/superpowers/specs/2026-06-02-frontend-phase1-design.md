# Frontend Phase 1 — Integration Foundation Design Spec

**Date:** 2026-06-02
**Status:** Draft, awaiting user review
**Scope:** First phase of the Expo React Native client overhaul — wire the existing screens to the real backend with a proper data/auth/state architecture.

## 1. Background & Motivation

The backend is complete across four merged sub-projects (Identity & Auth, Threads Core, Read Path, Moderation & Notifications) exposing ~10 controllers. The Expo client, however, is still the original scaffold: a clean visual prototype with **no data layer, no auth, and no integration** — every screen renders hardcoded mock data, and the login button calls `router.push('/(tabs)/home')` without authenticating.

Phase 1 turns the prototype into a real app: a typed API client, secure token storage with automatic refresh, TanStack Query for server state, SignalR for chat, and all of today's screens wired to real endpoints. New product surfaces the backend now supports (Threads, notifications, draw-card) are **out of scope** for Phase 1 — they have no screens yet and become later phases.

## 2. Goals & Non-Goals

**Goals**
- A layered integration architecture under `frontend/lib/` with clear, independently-testable units.
- Types generated from the backend's OpenAPI/Swagger so the client cannot silently drift from the contract.
- Secure token storage (`expo-secure-store`) + an auth interceptor implementing Plan 1's refresh-token rotation (401 → refresh-once → retry; single-flight; sign-out on refresh failure).
- TanStack Query for caching, invalidation, and consistent loading/empty/error handling.
- Wire ALL existing screens: login, register (+ activation), home/marketplace, product detail, sell (multipart upload), chat (SignalR realtime), profile, settings.
- Jest tests on the logic layer (client, interceptor, session, hooks) — no brittle screen-render tests.

**Non-Goals (later phases)**
- Threads feed/search, post/comment UI, anon toggle, likes.
- Notifications UI + unread badge.
- Moderation/admin UI.
- Draw-card feature.
- Deep-linking the email activation URL (dev uses the console-logged link).
- Cursor-based infinite scroll (marketplace list is flat today; arrives with the threads feed).
- Server-backed chat inbox list (no backend endpoint yet — see §7).

## 3. Architecture & Folder Structure

Layered, single-purpose units following the existing `@/*` alias + Expo Router conventions. Screens depend only on `query/hooks` + `auth-context`; hooks depend on `endpoints`; endpoints depend on `client` + generated `schema` types; `client` owns transport + the auth interceptor. Chat is the one exception (persistent socket) and lives in `realtime/`.

```
frontend/
  lib/
    config.ts              # API_BASE_URL from Expo config / EXPO_PUBLIC_API_URL
    api/
      schema.ts            # GENERATED by openapi-typescript (committed, never hand-edited)
      client.ts            # typed fetch wrapper: base URL, JSON/FormData, error normalization, Bearer header
      auth-interceptor.ts  # 401 → single-flight refresh → retry once; failure → clear session + emit signed-out
      endpoints.ts         # typed functions per call using schema.ts request/response types
    auth/
      session.ts           # secure-store {accessToken, refreshToken} + in-memory cache
      auth-context.tsx     # context: status loading|authed|anon, signIn, signOut, launch bootstrap
    query/
      query-client.ts      # TanStack Query client config
      keys.ts              # typed query-key factory
      hooks/               # one file per domain: items, item, categories, orders, me, reviews, register, createItem, chatHistory
    realtime/
      chat-connection.ts   # SignalR lifecycle (start/stop/reconnect, accessTokenFactory, ReceiveMessage)
    recent-chats.ts        # device-local recent conversations list (chat inbox, no backend endpoint yet)
  app/                     # existing screens rewired to hooks
```

**Codegen workflow:** npm script `gen:api` produces the backend OpenAPI JSON (`dotnet swagger tofile`, or curl the running `/swagger/v1/swagger.json`) and pipes it through `openapi-typescript` → `lib/api/schema.ts`. Committed artifact; re-run on backend contract changes. Because `endpoints.ts` imports its request/response types from `schema.ts`, a breaking backend DTO change surfaces as a TypeScript error at build time.

## 4. HTTP Client & Auth / Token Refresh

### session.ts
- `getSession()` / `setSession({accessToken, refreshToken})` / `clearSession()`.
- Tokens persist in `expo-secure-store` (encrypted) and mirror to a module-level in-memory cache so attaching the Bearer header is synchronous (no secure-store hit per request).
- On launch, `auth-context` reads secure-store once into the cache.

### client.ts
- Single `request<T>(path, opts)`: prepends `API_BASE_URL`, sets JSON headers (omits content-type for `FormData`), attaches `Authorization: Bearer <accessToken>` from the in-memory session, parses the body.
- **Error normalization:** backend returns `ProblemDetails` and `{ error }` shapes; both unwrap into one `ApiError { status: number, message: string, problem?: unknown }` so screens handle errors uniformly.

### auth-interceptor.ts
Wraps `request`. Flow:
```
request → 401?
  ├─ no  → return
  └─ yes → is the failing call /auth/refresh, or already retried once?
            ├─ yes → clearSession() + emit "signed-out" → return 401
            └─ no  → POST /api/auth/refresh { refreshToken }
                      ├─ ok   → setSession(NEW pair) → retry original ONCE
                      └─ fail → clearSession() + emit "signed-out"
```
Correctness details matching Plan 1:
- **Rotation:** refresh returns a *new* refresh token and revokes the old; the interceptor persists the whole new pair, not just the access token.
- **Single-flight:** concurrent 401s share one in-flight refresh promise, then each retries — otherwise parallel refreshes would rotate and invalidate each other. (Unit-tested.)

### auth-context.tsx
- `{ status: 'loading'|'authed'|'anon', signIn(email,password), signOut() }`.
- `signIn` → `POST /api/auth/login`, stores `{token, refreshToken, user}`, flips to `authed`.
- `signOut` → best-effort `POST /api/auth/logout` then `clearSession()` → `anon`.
- Listens for the interceptor's "signed-out" event to drop to `anon`.

### Navigation gating
Root `app/_layout.tsx` reads `status`: `loading`→splash; `anon`→redirect to `/` (login); `authed`→app. Replaces the fake `router.push` on the login button with a real `signIn()` that navigates only on success.

### Registration / activation
Plan 1 requires email activation before login. `register` → `POST /api/auth/register` → routes to a **"check your email to activate"** screen (not into the app). Dev: the activation link is logged to the backend console (documented). Deep-linking the activation URL is out of scope for Phase 1.

## 5. Server State (TanStack Query) & Per-Screen Wiring

`keys.ts` centralizes typed query keys (`items.all`, `items.detail(id)`, `categories.all`, `orders.mine`, `user.me`, `user.reviews(id)`, `chat.messages(otherUserId)`). Hooks wrap `endpoints` in `useQuery`/`useMutation`; mutations invalidate affected keys on success.

| Screen | Hook(s) → endpoint |
|---|---|
| `index.tsx` (login) | `auth-context.signIn` → `POST /api/auth/login` |
| `register.tsx` | `useRegister` → `POST /api/auth/register` → activation screen |
| `(tabs)/home.tsx` | `useItems` → `GET /api/items`; `useCategories` → `GET /api/categories`; search filters loaded list client-side |
| `product/[id].tsx` | `useItem(id)` → `GET /api/items/{id}`; Buy Now → `useCreateOrder` → `POST /api/orders`; Chat with seller → open 1:1 chat |
| `(tabs)/sell.tsx` | `useCreateItem` (multipart) → `POST /api/items`; `useCategories`; invalidate `items.all` on success |
| `(tabs)/profile.tsx` | `useMe` / `useUpdateProfile` → `GET`/`PATCH /api/users/me`; `useReviews(id)` → `GET /api/users/{id}/reviews` |
| `(tabs)/chat.tsx` | SignalR (§7) + `chat.messages` history; device-local recent list for the inbox |
| `settings.tsx` | `signOut`; notification/theme toggles stay local (no backend) |

**Reality checks (kept honest):**
- `GET /api/items` returns a flat list (no pagination) → home is a plain query; item search is client-side. Cursor infinite scroll is deferred to the threads feed (later phase).
- Optimistic updates are minimal in Phase 1 (no marketplace likes); create/update flows use invalidate-on-success. Optimistic UI is reserved for threads likes later.

**Loading/empty/error standardization:** a reusable pattern maps a `useQuery` result to spinner / empty-state / error-with-retry, applied consistently across data screens.

## 6. Multipart Upload (Sell)

`useCreateItem` builds `FormData` from metadata fields + images chosen via `expo-image-picker`, posts to `POST /api/items`. The client detects `FormData` and omits the JSON content-type so the multipart boundary is set by the runtime. Upload progress is best-effort (fetch lacks native progress) — acceptable for Phase 1.

## 7. Chat (SignalR Realtime)

Backend: hub at `/hubs/chat`, authenticated via an `access_token` **query param** (WebSockets can't send Authorization headers; the backend's `JwtBearerEvents` reads it for the hub path). Server method `SendMessage({ toUserId, body, itemId?, attachmentUrl? })`; client event `ReceiveMessage`. History: `GET /api/chats/{otherUserId}/messages` (1:1, keyed by other user).

`realtime/chat-connection.ts`:
- Builds a `HubConnection` with `accessTokenFactory: () => session.accessToken` and automatic reconnect; `start()` on entering chat, `stop()` on leaving.
- `ReceiveMessage` → append to the active thread; also reconcilable with React Query history.
- **Token-expiry handling:** on hub auth failure / disconnect, call the same refresh routine the interceptor uses, then reconnect with the fresh token.

**Known gap (documented):** the backend has no conversation-list endpoint — history is per specific other-user. Phase 1 therefore:
- Ships the working **1:1 thread** opened from "Chat with seller" (seller id known).
- Backs the **chat tab inbox** with a **device-local recent-conversations list** (`recent-chats.ts`, persisted on-device): each opened conversation is remembered locally so the inbox shows recent threads. This is explicitly device-only.
- **Backend follow-up (out of scope here):** add a `GET /api/chats` conversations-list endpoint to make the inbox server-backed and cross-device.

## 8. Error Handling

All errors normalize to `ApiError { status, message }`. Mapping:
- `401` → interceptor handles silently (refresh + retry); only a dead refresh reaches the UI as sign-out.
- `429` (login / report rate-limit) → friendly "try again shortly."
- `403` → "not allowed."
- `400` (validation) → surface the message.
- Network error → "check your connection," with retry.

## 9. Testing Strategy

Jest, logic layer only (no screen-render tests):
- `client`: header/body assembly, JSON vs FormData, error normalization of both backend error shapes.
- `auth-interceptor`: 401→refresh→retry success; single-flight under concurrent 401s; refresh-failure→signed-out; rotation persists the new pair; never loops on a 401 from `/auth/refresh` itself.
- `session`: secure-store round-trip + in-memory cache behavior (secure-store mocked).
- A couple of representative hooks with a mocked fetch: success populates cache; error surfaces `ApiError`; mutation invalidates the right key.

## 10. Config & Dependencies

- `API_BASE_URL` from Expo config (`app.json` `extra` / `EXPO_PUBLIC_API_URL`), defaulting to the local backend (`http://localhost:8080`); physical devices use the host LAN IP. README note on pointing the app at the backend + the dev activation-link caveat.
- New deps: `@tanstack/react-query`, `expo-secure-store`, `@microsoft/signalr`, `expo-image-picker`, `expo-image`; dev-only `openapi-typescript`.

## 11. Out of Scope (Future Phases)
- Threads (feed/search/post/comments/anon toggle/likes), notifications UI, moderation/admin UI, draw-card.
- Conversations-list backend endpoint + server-backed inbox.
- Activation deep-linking, push notifications, upload progress, infinite scroll.
